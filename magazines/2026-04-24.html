Supply chain · Breaking

Bitwarden CLI is the latest casualty of an ongoing npm supply chain campaign.

Socket's research team confirms that malicious versions of the Bitwarden CLI were published to npm as part of the Checkmarx-style worm that has now been tearing through the JavaScript ecosystem for weeks. The campaign keeps changing hands; each round poisons more trusted packages.

The compromised package behaves like the legitimate Bitwarden CLI on the surface, but exfiltrates environment variables, credentials, and the contents of developer secrets files to attacker-controlled infrastructure. Because the Bitwarden CLI is commonly invoked from CI pipelines and developer machines, a single curl-to-install or an npm upgrade inside a build image is enough to hand over the keys.

This is the same lineage of worm that previously hit Checkmarx, Vercel-adjacent OAuth tokens earlier in the month, and dozens of lesser-known packages. The pattern is depressingly familiar: an attacker steals maintainer credentials through phishing, publishes a subtly-malicious minor version, then uses the exfiltrated tokens to compromise more maintainers in the next wave.

What to do

Audit every build image, CI runner, and developer workstation for the Bitwarden CLI and pin to a known-good version. Rotate any secrets that have touched a machine running the CLI in the last fourteen days. Enforce npm package lockfiles with integrity hashes and disable auto-upgrades on CI. This is a good week to check whether your SBOM programme actually tells you which teams depend on this.

Surveillance · Investigation

Two surveillance vendors have been caught riding inside telco networks, silently geolocating targets worldwide.

TechCrunch's investigation details how commercial spyware operators piggybacked on legitimate telco interconnects to run bulk location tracking and silent-SMS attacks, reaching subscribers across dozens of carriers before being detected.

The findings echo the old SS7 and Diameter abuse playbooks but show a new level of sophistication: the operators rotated signalling identities between legitimate mobile networks, making each query look like routine roaming traffic. Victims included journalists, opposition politicians, and at least a handful of European civil servants, according to the researchers quoted.

The uncomfortable part for any CISO: if an attacker can buy passive geolocation and silent-SMS delivery as a service, then MFA via SMS is worse than it looked yesterday. Anything that depends on possession of a phone number without possession of the device is now in the blast radius.

Data protection · UK

UK Biobank genomic data keeps being uploaded to public GitHub repositories.

A running tracker at biobank.rocher.lc catalogues public repositories where researchers have uploaded fragments of UK Biobank data, apparently unaware that "anonymised" genomics on a public forge is not anonymised in any useful sense.

Most of the leaks appear accidental: intermediate files in analysis repos, config files that point at participant identifiers, or worked examples from PhD students. None of them individually violate the Biobank access agreement in a way that would have been caught by a naive scan, but collectively they chip away at the anonymity of an unusually rich dataset.

This matters for UK organisations in three ways. First: any research body with a Biobank access agreement should assume an audit is plausible in the next twelve months. Second: the ICO has been leaning harder on "reasonably likely to be identifiable" interpretations since the DUA Act came in. Third: GitHub scanning for PII and genomic markers is now a table-stakes control, not a nice-to-have.

What to do

If your organisation holds a UK Biobank access agreement, ask data protection whether you have an inventory of which repositories reference the dataset. Extend secret-scanning rules to cover Biobank participant identifier formats. Raise it with your DPO as a UK-specific exposure rather than a generic open-source hygiene item.

$ apt update · Infrastructure

Ubuntu 26.04 "Ferocious Fennec" lands, with stricter defaults.

The LTS cycle turns over today. LWN's release coverage walks through the changes that matter for fleet operators: a stricter AppArmor baseline, the official move to dm-verity for the system partition on server images, and a rethought unattended-upgrades path.

Notable for anyone running containers at scale: the default Python interpreter is now 3.13, and OpenSSL has been rebased on 3.4 with stricter protocol defaults. A handful of older TLS suites that were still compiled in as opt-in are now compile-time disabled.

The boring but important bit: the upgrade path from 24.04 LTS is clean on tested paths but the server installer has been reworked, so expect some automation around PXE and cloud-init to need tweaks. Canonical is promising ten years of security updates on the server SKU, same as the last cycle.

Give it a fortnight in a lab before you bless it for production.

AI tooling · Postmortem

Anthropic explains what went wrong with Claude Code in April.

The company has published a detailed postmortem on the wave of quality reports that hit Claude Code users earlier this month, and it reads like a serious piece of engineering writing rather than PR cover.

The short version: an interaction between a new inference kernel, a context-caching optimisation, and a late change to the tool-use prompt template caused regressions that were invisible in evals but visible in the shape of real developer sessions. Anthropic notes that their internal benchmarks were overfit to short-turn tasks and missed regressions on the longer, tool-heavy workflows that production users actually run.

There is a broader lesson buried in the document. For anyone whose team uses Claude Code or Cowork day-to-day, the takeaway is that model quality is now a supply chain concern: a silent regression in a provider's inference stack can show up as a productivity tax on your engineers before the provider notices it in their own metrics. Anthropic has added new evals, widened canary rollouts, and committed to a faster rollback path. Worth reading in full.

What to do

Read the postmortem. If your team leans on Claude Code, consider a lightweight internal "how's it feeling today" pulse and a standing rollback plan to an older model alias. Treat model provider changes as change-managed events, not silent upgrades.

AI model · Launch

OpenAI ships GPT-5.5 with a longer context and a cheaper tier.

A mid-cycle bump rather than a full version jump: GPT-5.5 lands with a larger context window, a new mini tier aimed at agentic workloads, and the usual round of reasoning benchmarks that ship alongside any OpenAI release.

On paper, GPT-5.5 narrows the gap with the current top of the Claude and Gemini ladders on long-context reasoning and tool use. OpenAI has been emphasising the mini tier, which is priced aggressively for agent workloads that make a lot of small tool calls, and a new low-latency mode for voice that is likely where the real product bet is sitting.

For the CISO seat: the interesting footnote is a tightened default for data-retention on the API, with a clearer opt-out for log retention that matters in regulated industries. The more interesting second-order effect is what this does to competitive pricing in the agent space over the next quarter.

Breach · France

French government agency confirms breach as hacker offers to sell the data.

BleepingComputer reports a confirmed intrusion at a French government agency, with the attacker already advertising the dataset for sale on a Russian-speaking forum. Officials have notified the data protection authority and activated incident protocols.

The agency has not published full technical details, but the initial notice suggests an externally-exposed web application and a credential theft leading to access to an internal document store. The advertised dataset appears to include employee directories, internal forms, and correspondence, though not the kind of citizen-facing records that would trigger a mass-notification event.

What makes this one worth watching is less the incident itself and more what it signals for the French and wider European public sector. After the wave of breaches at ministries and health bodies over the last year, the question is no longer whether a given agency will be hit but whether it has the instrumentation to find out quickly. CNIL has been clear that "we found out when the data appeared on a forum" is not acceptable as a default detection path.

Expect the details to firm up over the next few days as the agency works through notifications. As ever with these stories, the second attacker wave — the one that buys the dataset and uses it for targeted phishing — is where the real damage happens.

AI security · Open source

Agent Vault: an open-source credential proxy for AI agents.

Infisical has shipped Agent Vault, an open-source proxy that sits between an AI agent and the credentials it needs to call tools. The agent never sees the secret; the proxy injects it at request time and logs what was used.

This is the shape of the answer most security teams have been waiting for when engineers hand them an agentic workflow and say "but it needs the API keys". The proxy pattern lets you audit what an agent did with which credential, scope credentials per tool call, and pull a secret out of circulation without restarting anything. The project is MIT-licensed and includes adapters for common agent frameworks.

It is worth treating the release as a reference design rather than a drop-in fix. The assumptions it makes about how agents run, how tools are discovered, and how identity is propagated will not all match your environment, but the ideas are the right ones. For anyone who is building Claude Code or Cowork workflows and has been hand-waving the credential story, this is a good starting read.

What to do

Spend twenty minutes on the repository and the architecture diagram. Use it as a reference when your team proposes the next agentic automation. At minimum, steal the logging schema and the per-tool-call scoping primitive.

Policy · Europe

Armin Ronacher on why equity doesn't work the way it should for European founders.

A blog post that has been doing the rounds on Hacker News walks through, with real numbers, why the same equity grant means something radically different depending on whether you are sitting in California or anywhere in the EU or the UK.

The headline issue is tax treatment at exercise and sale, but the more interesting observation is about expectations. European tech workers negotiate equity as if it were a nice-to-have because, historically, it has paid out worse than cash, which means European companies can get away with granting less of it, which means the equity-as-wealth-creator story never really compounds the way it does in the US. It is a coordination problem dressed up as tax policy.

The post is not a policy prescription; it is a quietly devastating tour of why the usual "just use SARs" or "just use phantom stock" answers do not actually fix the underlying asymmetry. Worth reading if you hire technical talent in the UK or EU, or if you think about where the next wave of European defensive-tech companies will be founded.

Ethics · Inside Palantir

Some Palantir engineers are asking, quietly, whether they are the bad guys.

Wired talks to a handful of current and recent Palantir employees about the moral drift they describe feeling as the company's work with ICE and domestic surveillance programmes has expanded.

The interviews land softly rather than as exposé. Nobody quoted is a whistleblower; most of them are people who joined believing the "we will build ethical software for hard problems" pitch and are now less sure. Some have left. Some are staying. The piece is good on the psychology of that drift, and less good on the strictly legal question of what Palantir is and is not contracted to do.

For security leaders, this is adjacent-not-central reading, but it is worth the time. The engineers describing their dissonance sound a lot like the people who end up in security ethics conversations five years later. It is also a reminder that vendor selection is, among other things, a values statement that will be read by the technical staff you want to keep.
